## 第一部分：啟動完整 infra（RabbitMQ + MySQL）

EP06 只開了 RabbitMQ。EP07 要同時開 RabbitMQ（訊息佇列）和 MySQL（資料庫），
所以這次用的是「帶網路」的版本。

### Step 1：建立共用 Docker 網路

多個 Compose 檔（RabbitMQ、MySQL、Worker）要互相通訊，必須掛在**同一個網路**上。
這些 `*-network.yml` 都用 `external: true`，代表網路要**先手動建好**。

```bash
docker network create my_network
```

✅ **預期**：印出一段網路 ID（例如 `a1b2c3d4...`）

```bash
# 確認網路已存在
docker network ls | grep my_network
```

> 💡 如果出現 `network with name my_network already exists`，代表之前建過了，直接略過即可。

### Step 2：啟動 RabbitMQ + Flower

```bash
cd ~/stock-crawler
docker compose -f rabbitmq-network.yml up -d
```

✅ **預期**：`rabbitmq` 和 `flower` 兩個服務都 Up

```bash
docker compose -f rabbitmq-network.yml ps
```

### Step 3：啟動 MySQL + phpMyAdmin

```bash
docker compose -f mysql.yml up -d
```

✅ **預期**：`mysql` 和 `phpmyadmin` 兩個服務都 Up

```bash
docker compose -f mysql.yml ps
```

### Step 4：確認所有 Web 介面都通

```bash
curl -s -o /dev/null -w "RabbitMQ: %{http_code}\n" http://localhost:15672   # 預期 200
curl -s -o /dev/null -w "Flower:   %{http_code}\n" http://localhost:5555    # 預期 200
curl -s -o /dev/null -w "phpMyAdmin: %{http_code}\n" http://localhost:8080  # 預期 200
```

| 服務 | 網址 | 帳密 |
|------|------|------|
| RabbitMQ 管理介面 | http://localhost:15672 | worker / worker |
| Flower 監控 | http://localhost:5555 | （無） |
| phpMyAdmin | http://localhost:8080 | user / ppWgnb_mfGe2m_ |

> ⚠️ MySQL 第一次啟動需要幾秒鐘初始化，如果 phpMyAdmin 連不上，等 10 秒再 curl 一次。

## 第二部分：多 Worker Compose（twse + tpex）

EP06 只有一個 Worker。一旦任務變多，它只能一個一個慢慢做。
EP07 用 `docker-compose-worker-network.yml` 一次開**兩個 Worker**：

- `crawler_twse`：專門處理 `twse`（證交所，上市）佇列
- `crawler_tpex`：專門處理 `tpex`（櫃買中心，上櫃）佇列

### Step 1：看懂 Worker Compose

```bash
cat docker-compose-worker-network.yml
```

重點在每個服務的 `command`：

```yaml
command: uv run celery -A crawler.worker worker --loglevel=info --hostname=%h -Q twse
```

| 參數 | 意義 |
|------|------|
| `-A crawler.worker` | 指定 Celery app 在哪 |
| `--hostname=%h` | 用容器 hostname 當 worker 名稱（twse / tpex） |
| `-Q twse` | 這個 worker **只**消費 `twse` 佇列的任務 |

兩個服務都設了 `RABBITMQ_HOST=rabbitmq`、`MYSQL_HOST=mysql`，
並掛在 `my_network` 上，才能找到第一部分啟動的 RabbitMQ 和 MySQL。

### Step 2：啟動兩個 Worker

```bash
docker compose -f docker-compose-worker-network.yml up -d
```

✅ **預期**：`crawler_twse` 和 `crawler_tpex` 兩個服務都 Up

```bash
docker compose -f docker-compose-worker-network.yml ps
```

### Step 3：看 Worker log 確認 ready

```bash
# 看 twse worker
docker compose -f docker-compose-worker-network.yml logs crawler_twse | tail -20
```

✅ **預期**：看到 Celery 啟動橫幅，最後一行類似

```
celery@twse ready.
[queues]
  .> twse   exchange=twse(direct) key=twse
```

```bash
# 看 tpex worker
docker compose -f docker-compose-worker-network.yml logs crawler_tpex | tail -20
```

✅ **預期**：`celery@tpex ready.`，且 queues 是 `tpex`。

> 💡 **進階：用版本變數啟動**
> 若要指定映像檔版本，改用 `-version` 那支 compose（需先準備 `.env`）：
> ```bash
> DOCKER_IMAGE_VERSION=0.0.6 docker compose -f docker-compose-worker-network-version.yml up -d
> ```
> 一般練習用上面 Step 2 的 `docker-compose-worker-network.yml` 即可，不需這步。

## 第三部分：跑 FinMind 真實爬蟲

infra 和 Worker 都備好了，現在來發**真實的 FinMind 爬蟲任務**。
先用「只印出、不寫 DB」的 print 版確認流程，再用正式版。

### Step 1：先發 print 版（不寫 DB）

`producer_crawler_finmind_print.py` 用 for 迴圈一次發 5 支股票：

```bash
cat crawler/producer_crawler_finmind_print.py
```

```python
from crawler.tasks_crawler_finmind import crawler_finmind_print

for stock_id in ["2330", "0050", "2317", "0056", "00713"]:
    print(stock_id)
    crawler_finmind_print.delay(stock_id=stock_id)
```

執行：

```bash
uv run python crawler/producer_crawler_finmind_print.py
```

✅ **預期**：依序印出 `2330 / 0050 / 2317 / 0056 / 00713`，程式秒退（任務已丟進 RabbitMQ）。

> ⚠️ 注意：這個 print 版**沒指定 queue**，所以任務送到 Celery 預設的 `celery` queue。
> 第二部分啟動的 worker 只聽 `twse` / `tpex`，**不會**消費到預設 queue。
> 想看到結果，第五部分的「多 Queue 分流」才會把任務正確路由到 twse / tpex worker。
> 這一步的目的是先確認 **producer 發得出去**（看 RabbitMQ 佇列有沒有堆積）。

驗證任務有進 RabbitMQ：

```bash
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:15672
# 開瀏覽器到 http://localhost:15672 → Queues 分頁 → 看到 celery queue 有 5 則訊息堆積
```

### Step 2：看懂正式版 producer

`producer_crawler_finmind.py` 是會寫進 MySQL 的正式版：

```bash
cat crawler/producer_crawler_finmind.py
```

它一樣用 for 迴圈發任務，但對應的 task 是 `crawler_finmind`（抓資料 + 寫 MySQL），
而非 print 版。完整的 DB 寫入驗證留到 **EP08** 再深入。

## 第四部分：用 Flower 監控多 Worker

Flower 是 Celery 的 Web 監控介面，能即時看到「有幾個 Worker、誰在做什麼、成功還是失敗」。

### Step 1：打開 Flower

瀏覽器開 http://localhost:5555

### Step 2：看 Workers 分頁

✅ **預期**：看到 **兩個 worker** 上線：

| Worker name | Status | Queues |
|-------------|--------|--------|
| celery@twse | Online | twse |
| celery@tpex | Online | tpex |

這就是「水平擴展」最直觀的證據——同一份程式碼，跑了兩個獨立的 Worker。

### Step 3：看 Tasks 分頁

跑完第五部分發任務後，回來這個分頁，可以看到每個 task 的：
- **State**：SUCCESS（成功）/ FAILURE（失敗）/ STARTED（執行中）
- **Worker**：是哪個 Worker 接手的（twse 還是 tpex）
- **Runtime**：執行花了幾秒

> 💡 多 Worker 的重點就是看 task **被分散到不同 Worker**，而不是全擠在一個身上。

## 第五部分：多 Queue 分流

預設情況下所有任務都排在同一條 queue。實務上常要依「任務類型」分流——
上市股票走 `twse`、上櫃/特殊標的走 `tpex`，各自由專屬 Worker 處理，避免互搶資源。

### Step 1：看懂 multi queue producer

```bash
cat crawler/producer_multi_queue.py
```

```python
from crawler.tasks_crawler_finmind import crawler_finmind

# .s() = signature，把 task 和參數綁成可派送物件
task_2330 = crawler_finmind.s(stock_id="2330")
task_2330.apply_async(queue="twse")    # 指定送到 twse queue
print("send task_2330 task")

task_00679b = crawler_finmind.s(stock_id="00679B")
task_00679b.apply_async(queue="tpex")  # 指定送到 tpex queue
print("send task_00679b task")
```

關鍵差異：

| 寫法 | 行為 |
|------|------|
| `.delay(stock_id=...)` | 送到**預設** queue |
| `.s(...).apply_async(queue="twse")` | **指定** queue，由對應 `-Q twse` 的 worker 消費 |

### Step 2：發多 queue 任務

```bash
uv run python crawler/producer_multi_queue.py
```

✅ **預期**：印出

```
send task_2330 task
send task_00679b task
```

### Step 3：驗證分流結果（看 Worker log）

```bash
# 2330（上市）應該被 twse worker 接走
docker compose -f docker-compose-worker-network.yml logs crawler_twse | tail -15

# 00679B 送到 tpex queue，應該被 tpex worker 接走
docker compose -f docker-compose-worker-network.yml logs crawler_tpex | tail -15
```

✅ **預期**：
- `crawler_twse` log 出現 `Task crawler.tasks_crawler_finmind.crawler_finmind ... succeeded`，且資料是 2330。
- `crawler_tpex` log 出現對應的 succeeded，資料是 00679B。

### Step 4：到 Flower 對照

回 http://localhost:5555 → Tasks 分頁：

✅ **預期**：兩筆 task，State 都是 **SUCCESS**，且 Worker 欄位一個是 `celery@twse`、一個是 `celery@tpex`。

> 💡 這就是「多 Queue + 多 Worker」的完整閉環：producer 指定 queue → RabbitMQ 路由 → 專屬 worker 消費。

> 🐛 **常見問題**：如果 task 卡在 RECEIVED / 沒人做，多半是「queue 名稱對不上」——
> 確認 producer 的 `queue="twse"` 和 worker 的 `-Q twse` 完全一致（大小寫也要一樣）。

## 第六部分：Unit Test 入門（pytest）

爬蟲跑得起來只是第一步。工程上還要**自動驗證**——程式改壞了能馬上被測出來。
這部分用 `pytest` 寫第一個測試，驗證 `config.py` 的預設值正確。

### Step 1：認識 pytest 的規則

pytest 會自動尋找並執行：
- 檔名 `test_*.py` 或 `*_test.py`
- 函式名 `test_*`
- 用 `assert` 判斷對錯，成立就 PASS，不成立就 FAIL

### Step 2：寫一個 config 測試

專案根目錄的 `test_config.py` 原本只是「印出設定」的 smoke 腳本。
我們把它改成**真正的 pytest 測試**。用編輯器打開 `test_config.py`，改成：

```python
# test_config.py
from crawler.config import WORKER_ACCOUNT


def test_default_account():
    # config.py 沒被環境變數覆蓋時，預設帳號應該是 "worker"
    assert WORKER_ACCOUNT == "worker"
```

> 💡 為什麼測這個？`config.py` 是整個專案的「設定中樞」，預設值錯了，
> 本機開發連 RabbitMQ 就會失敗。用一個小測試把它釘住。

### Step 3：執行測試

```bash
uv run pytest test_config.py -v
```

✅ **預期**：

```
test_config.py::test_default_account PASSED                    [100%]

============================== 1 passed in 0.0Xs ===============================
```

看到 **1 passed** 和綠色 `PASSED` = 測試通過。

### Step 4：故意弄錯，看測試抓到（RED）

把 assert 改成錯的值，體驗測試「失敗時長什麼樣」：

```python
def test_default_account():
    assert WORKER_ACCOUNT == "admin"   # 故意寫錯
```

```bash
uv run pytest test_config.py -v
```

✅ **預期**：紅色 `FAILED`，並指出

```
E       AssertionError: assert 'worker' == 'admin'
```

這就是測試的價值——它替你盯著「值對不對」。看完記得改回 `"worker"` 讓它變綠。

### Step 5：跑整個專案的測試（選做）

```bash
uv run pytest -v
```

pytest 會掃描整個專案、跑所有 `test_*.py`。日後新增功能就新增對應測試，
養成「寫程式同時寫測試」的習慣。
