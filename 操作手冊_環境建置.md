## 第一段：作業系統環境準備（EP01）

### Windows：安裝 WSL + Ubuntu

#### 為什麼要用 WSL？

- 業界開發和部署都用 Linux 環境
- 安裝程式語言、開源套件在 Linux 上最方便
- macOS 本身是 Unix-like，指令跟 Linux 幾乎一樣，所以不用另外裝
- WSL（Windows Subsystem for Linux）讓你在 Windows 裡直接跑 Linux

#### Step 1：確認虛擬化已開啟

🪟 Windows

```
1. 按 Ctrl+Shift+Esc 開啟工作管理員
2. 點「效能」→「CPU」
3. 右下角找「虛擬化」→ 確認顯示「已啟用」
```

✅ **驗證**：看到「虛擬化：已啟用」

⚠️ 如果顯示「已停用」→ 需要進 BIOS 開啟（見 [FAQ：WSL Q1](#q1wsl---install-出現錯誤-0x80370102)）

#### Step 2：開啟 Windows 功能

🪟 Windows

```
1. 控制台 → 程式與功能 → 開啟或關閉 Windows 功能
2. 勾選：
   ☑ Windows 子系統 Linux 版
   ☑ 虛擬機器平台（Virtual Machine Platform）
3. 點確定，等待安裝
4. 重新啟動電腦
```

⚠️ **不需要勾 Hyper-V**。WSL2 只需要「虛擬機器平台」，`wsl --install` 也會自動啟用。Windows Home 版本身就沒有 Hyper-V，完全不影響。

#### Step 3：安裝 WSL + Ubuntu

🪟 重開機後，Windows PowerShell（系統管理員）

```powershell
# 📁 任意目錄

# 方法 A：一行安裝（推薦）
wsl --install -d Ubuntu-22.04

# 方法 B：如果 Store 有問題，分步安裝
wsl --install --no-distribution
# 重開機後再跑：
wsl --install -d Ubuntu-22.04
```

✅ **驗證**：

```powershell
wsl -l -v
# 預期輸出：
#   NAME            STATE           VERSION
# * Ubuntu-22.04    Running         2
```

⚠️ VERSION 必須是 **2**（不是 1）。如果是 1，見 [FAQ：WSL Q2](#q2wsl--l--v-顯示-version-是-1-不是-2)

#### Step 4：設定 WSL2 為預設

🪟 Windows PowerShell

```powershell
wsl --set-default-version 2
```

#### Step 5：第一次開啟 Ubuntu

🪟 從開始選單搜尋 **Ubuntu 22.04**，點開

```
1. 第一次開啟會跑初始化（約 1-2 分鐘）
2. 看到 Enter new UNIX username → 輸入小寫英文帳號（例如 dex）
3. New password → 輸入密碼（畫面不會顯示任何字元，這是正常的）
4. Retype new password → 再輸入一次確認
5. 看到 $ 提示符號 → 成功！
```

⚠️ **username 必須是英文小寫**，不能有空格或中文

✅ **驗證**：

```bash
# 📁 /home/你的帳號

# 確認 Ubuntu 版本
cat /etc/os-release
# 預期：NAME="Ubuntu"  VERSION_ID="22.04"

# 確認是否為你的帳號（不是 root）
whoami
# 預期：你的帳號名稱（例如 dex）
```

#### Step 6：更新系統 + 確認工具

🐧 WSL Ubuntu Terminal

```bash
# 📁 /home/你的帳號

# 更新系統套件
sudo apt update && sudo apt upgrade -y

# 確認 Python
python3 --version
# 預期：Python 3.10.x

# 安裝 pip
sudo apt install python3-pip -y
pip3 --version
# 預期：pip 22.x.x

# 確認 Git
git --version
# 預期：git version 2.x.x
# 如果 command not found：
sudo apt install git -y
```

✅ **驗證清單**（三個都要有版本號）：

```
python3 --version  → Python 3.10.x  ✅
pip3 --version     → pip 22.x.x     ✅
git --version      → git 2.x.x      ✅
```

### 常見狀況：開啟 WSL 後發現在 /mnt/c/Users/...

如果你打開 Ubuntu 後 `pwd` 顯示 `/mnt/c/Users/XXX`，代表你在 Windows 的目錄下（`C:\Users\XXX` → `/mnt/c/Users/XXX`），不是 Linux 原生目錄。

```bash
# 1. 確認目前位置
pwd
# 如果顯示 /mnt/c/Users/XXX → 你在 Windows 目錄下

# 2. 回到 Linux 家目錄
cd ~

# 3. 確認已經回到正確位置
pwd
# 應顯示 /home/你的帳號
```

⚠️ **如果顯示 /root**：代表你用 root 登入，需要建立一般使用者：

```bash
# 在 WSL 裡（root 身份）
adduser 你的帳號
usermod -aG sudo 你的帳號
```

然後回到 Windows CMD / PowerShell 設定預設使用者：

```powershell
# 先關掉 WSL
wsl --shutdown
# 設定預設使用者
ubuntu2204 config --default-user 你的帳號
# 重新開啟 Ubuntu
```

> **重要觀念**：
> - `/mnt/c/...` = Windows 檔案系統，存取速度慢，權限可能有問題
> - `/home/你的帳號/` = Linux 原生目錄，**開發統一在這邊**
> - 所有課程操作都在 `/home/你的帳號/de-01-projects/` 底下進行

## 第二段：Linux 基礎指令（EP01）

以下指令在 **WSL Ubuntu Terminal** 或 **macOS Terminal** 操作。

### 導航與檔案操作

🐧 Terminal

```bash
# 📁 ~ (home 目錄)

# 看你在哪個目錄
pwd
# Windows WSL 預期：/home/你的帳號
# Mac 預期：/Users/你的帳號

# 看目錄內容
ls
ls -la        # 含隱藏檔、詳細資訊

# 建立資料夾
mkdir de-01-projects

# 進入資料夾
cd de-01-projects
pwd
# 預期：/home/你的帳號/de-01-projects（WSL）
# 或：/Users/你的帳號/de-01-projects（Mac）

# 建立檔案
touch hello.txt

# 寫入內容
echo "Hello from WSL" > hello.txt

# 讀取檔案
cat hello.txt
# 預期：Hello from WSL

# 複製檔案
cp hello.txt backup.txt

# 改名 / 移動
mv backup.txt renamed.txt

# 刪除檔案（不進垃圾桶，無法復原！）
rm renamed.txt

# 確認只剩 hello.txt
ls
```

### 常用指令速查

| 指令 | 功能 | 範例 |
|------|------|------|
| `pwd` | 顯示當前目錄 | `pwd` |
| `ls` | 列出目錄內容 | `ls -la` |
| `cd` | 切換目錄 | `cd ~/projects` |
| `mkdir` | 建立資料夾 | `mkdir my-folder` |
| `touch` | 建立空檔案 | `touch test.py` |
| `echo` | 輸出文字 / 寫入檔案 | `echo "hi" > file.txt` |
| `cat` | 讀取檔案內容 | `cat hello.txt` |
| `cp` | 複製 | `cp a.txt b.txt` |
| `mv` | 移動 / 改名 | `mv old.txt new.txt` |
| `rm` | 刪除檔案 | `rm file.txt` |
| `nano` | 編輯檔案 | `nano test.txt`（Ctrl+O 存、Ctrl+X 離開） |
| `htop` | 活動監視器 | `htop`（按 q 離開） |

✅ **驗證**：能成功執行 `cat hello.txt` 並看到內容

## 第四段：Git 與 GitHub（EP01-2）

### Git 基礎概念

```
Working Directory ── git add ──→ Staging Area ── git commit ──→ Repository
  你正在編輯的檔案         暫存區（準備提交）           版本歷史
```

- **Git** = 版本控制工具（本地）
- **GitHub** = 程式碼託管平台（雲端）
- **origin** = remote 的預設名稱（git clone 自動設定）

### Git 初始設定

🐧 Terminal

```bash
# 設定使用者名稱和 email（只需設定一次）
git config --global user.name "你的 GitHub 顯示名稱"
git config --global user.email "你的 GitHub email"

# 確認設定
git config --list
# 預期看到 user.name 和 user.email
```

✅ **驗證**：`git config --global user.name` 有輸出你的名字

### 方法一：用 Terminal 指令操作 Git

🐧 Terminal

```bash
# 📁 ~/de-01-projects

# 1. 建立專案資料夾
mkdir my-team-project
cd my-team-project

# 2. 初始化 Git
git init
# 預期：Initialized empty Git repository in ...

# 3. 建立第一個檔案
echo "# My Team Project" > README.md
ls
# 預期：README.md

# 4. 查看狀態（紅色 = 未追蹤）
git status
# 預期：Untracked files: README.md

# 5. 加入暫存區（紅色 → 綠色）
git add README.md
git status
# 預期：Changes to be committed: README.md

# 6. 建立第一個 commit
git commit -m "init project readme"
# 預期：1 file changed, 1 insertion(+)

# 7. 看歷史
git log --oneline
# 預期：一行 commit hash + 訊息
```

### 連結 GitHub + Push

```bash
# 1. 先在 GitHub 網站建立空白 Repo（不要勾 README、.gitignore）
# 2. 回到 Terminal

# 設定主分支名稱
git branch -M main

# 連結遠端
git remote add origin https://github.com/你的帳號/my-team-project.git

# 推送（第一次需要認證）
git push -u origin main
```

✅ **驗證**：打開 GitHub repo 頁面，看到 README.md 和 commit 歷史

### git clone：下載別人的專案

```bash
# 📁 ~/de-01-projects

git clone https://github.com/DataEngCamp/test
cd test
ls
# 預期：看到專案裡的檔案
```

### 方法二：用 VS Code GUI 操作 Git

```
1. 左側 Source Control 圖示（或 Ctrl+Shift+G）
2. 按 Initialize Repository = git init
3. 修改檔案後 → 檔案出現在 Changes 區
4. 點檔案旁的 + 號 = git add
5. 上方輸入 commit message → 按 ✓ 或 Ctrl+Enter = git commit
6. 按 Publish Branch = push 到 GitHub
7. Git Graph 按鈕 = 看 commit 時間線
```

✅ **驗證**：Git Graph 能看到至少一個 commit 點

### .gitignore

有些檔案不要上傳到 GitHub：

```bash
# 建立 .gitignore
cat > .gitignore << 'EOF'
# 環境
```

### 完整的新專案上傳 GitHub 流程

```bash
# 1. 在 GitHub 建空白 Repo（不要勾 README、.gitignore）
# 2. 本機 Terminal：
cd /path/to/project
git init
git add .
git commit -m "initial commit"
git branch -M main
git remote add origin <repo-url>
git push -u origin main
```

### 常用 Git 指令速查

| 指令 | 功能 |
|------|------|
| `git init` | 初始化新的 Git repo |
| `git clone <url>` | 從遠端下載專案 |
| `git status` | 查看檔案狀態 |
| `git add <file>` | 加入暫存區 |
| `git add .` | 加入所有變更 |
| `git commit -m "msg"` | 建立版本 |
| `git log --oneline` | 查看精簡歷史 |
| `git push` | 上傳到遠端 |
| `git pull` | 從遠端拉取更新 |
| `git fetch` | 抓取遠端更新（不合併） |
| `git diff` | 看檔案差異 |
| `git branch` | 列出分支 |
| `git checkout -b <name>` | 建立並切換分支 |
| `git merge <branch>` | 合併分支 |

## 第六段：uv 環境管理（EP02）

### 為什麼需要環境管理？

- 專案 A 需要 Python 3.9 + requests 2.28
- 專案 B 需要 Python 3.12 + requests 2.31
- 全域安裝 → 版本衝突、互相干擾
- 不同專案需要不同版本的同一個套件
- 團隊協作時「我的電腦可以跑」的問題

💡 **環境管理 = 讓每個專案有自己獨立的 Python + 套件，互不干擾**

### 環境管理工具比較

過去的方案各有缺點：
- ✗ pyenv + virtualenv + pip = **三個工具**才完成一件事
- ✗ pipenv 安裝速度慢（有時要等幾分鐘）
- ✗ conda 太重，非資料科學用不到
- ✗ 版本切換容易搞混，忘記 activate
- ✗ requirements.txt 沒有鎖版，環境不可重現

| 工具 | Python 版本 | 虛擬環境 | 套件管理 | 速度 | 學習成本 |
|------|:---------:|:------:|:------:|:----:|:------:|
| pyenv + venv + pip | ✓ | ✓ | ✓ | 普通 | 高（三工具） |
| pipenv | ✗ | ✓ | ✓ | 慢 | 中 |
| conda | ✓ | ✓ | ✓ | 慢 | 高 |
| **uv** ⭐ | ✓ | ✓ | ✓ | **極快** | **低** |

### 為什麼直接用 uv？

- **Rust 寫的**，速度是 pip 的 10-100 倍 ⚡
- **一個工具** = Python 版本 + 虛擬環境 + 套件管理
- 支援 `pyproject.toml`（現代 Python 標準）
- 自動產生 `uv.lock` 鎖版檔，確保環境可重現
- 一句話：**uv 就是 Python 版的 nvm + npm**

### uv vs 傳統工具對照

| 功能 | 傳統做法 | uv 做法 |
|------|---------|--------|
| 安裝 Python | `pyenv install 3.12` | `uv python install 3.12` |
| 建虛擬環境 | `python -m venv .venv` | `uv venv` |
| 啟動虛擬環境 | `source .venv/bin/activate` | **不需要**（uv run 自動處理） |
| 安裝套件 | `pip install requests` | `uv add requests` |
| 鎖版 | `pip freeze > requirements.txt` | 自動產生 `uv.lock` |
| 執行程式 | `python hello.py` | `uv run python hello.py` |
| 專案設定檔 | `requirements.txt` | `pyproject.toml` |

## 第七段：Module / Package（EP02）

### 什麼是 Module？

- **一個 .py 檔 = 一個 Module**
- 可以被其他 .py `import` 來使用
- 例如：`ptt_crawler.py` 就是一個 module

### 問題：import 會執行整個檔案

```python
# ptt_crawler.py
import requests
from bs4 import BeautifulSoup

def ptt_crawler(board):
    url = f"https://www.ptt.cc/bbs/{board}/index.html"
    res = requests.get(url, headers={"cookie": "over18=1"})
    soup = BeautifulSoup(res.text, 'html.parser')
    titles = [t.text for t in soup.select('.title a')]
    return titles

# ↓↓↓ 注意這兩行 ↓↓↓
result = ptt_crawler("Stock")    # ← import 時也會跑！
print(result)                     # ← import 時也會跑！
```

當你在 `main.py` 裡寫 `from ptt_crawler import ptt_crawler`：
- Python 會從頭到尾執行 `ptt_crawler.py`
- `def` 和 `class` → 只定義不執行 ✅
- 不在 `def` / `class` 裡的程式碼 → **直接執行** ❌

### 解法：if \_\_name\_\_ == '\_\_main\_\_'

```python
# ptt_crawler.py（修改後）
def ptt_crawler(board):
    # ... 同前 ...
    return titles

if __name__ == "__main__":
    result = ptt_crawler("Stock")
    print(result)
```

原理：

| 執行方式 | `__name__` 的值 | 底部程式碼 |
|---------|-----------------|-----------|
| `python ptt_crawler.py`（直接跑） | `"__main__"` | ✅ 會執行 |
| `from ptt_crawler import ...`（被 import） | `"ptt_crawler"` | ❌ 不執行 |

### 實際操作驗證

🐧 Terminal（VS Code 內建）

```bash
# 📁 ~/de-01-projects/de-course/ep02

# 測試 package import（不會自動跑爬蟲）
uv run python -c "from crawlers import ptt_crawler; print(type(ptt_crawler))"
# 預期：<class 'function'>  ← 沒有印出爬蟲結果，因為 if __name__ 生效

# 手動呼叫函式
uv run python -c "
from crawlers import ptt_crawler
result = ptt_crawler('Tech_Job')
print(f'爬到 {len(result)} 篇文章')
"
# 預期：爬到 N 篇文章
```

### 什麼是 Package？

- **Package = 資料夾 + `__init__.py`**
- 用資料夾整理多個 module
- `__init__.py` 決定「對外暴露什麼」

#### 目錄結構

```
de-course/ep02/
├── crawlers/              ← Package！
│   ├── __init__.py        ← 關鍵檔案
│   ├── ptt_crawler.py
│   ├── hahow_crawler.py
│   └── finmind.py
├── ptt_crawler.py         ← 原始版（沒有 if __name__）
├── pyproject.toml
└── uv.lock
```

### \_\_init\_\_.py 怎麼寫

```python
# crawlers/__init__.py
from .ptt_crawler import ptt_crawler
from .finmind import crawler_finmind
from .hahow_crawler import crawler_hahow_course
```

- `.` = 相對匯入，表示「從同一個資料夾裡的」
- 效果：外面可以直接 `from crawlers import ptt_crawler` 拿到函式

### 有 vs 沒有 \_\_init\_\_.py

| 比較項目 | 沒有 \_\_init\_\_.py | 有 \_\_init\_\_.py |
|---------|---------------------|-------------------|
| `from crawlers import ptt_crawler` | 拿到 **module** 物件 | 直接拿到 **function** |
| 使用方式 | `ptt_crawler.ptt_crawler()` | `ptt_crawler()` |
| `dir(crawlers)` | 空的 `[]` | 列出所有 exports |
| 控制暴露 | 全部都可存取 | 只暴露指定的 |

### 建 Package 的步驟（從零開始）

```bash
# 1. 建資料夾
mkdir crawlers

# 2. 複製 .py 進去（每個要加 if __name__）
cp ptt_crawler.py crawlers/
cp hahow_crawler.py crawlers/
cp finmind.py crawlers/

# 3. 建立 __init__.py
touch crawlers/__init__.py

# 4. 編輯 __init__.py，寫入 from 語句

# 5. 測試
python -c "from crawlers import ptt_crawler; print(type(ptt_crawler))"
# 預期：<class 'function'>
```

### 從 Level 1 到 Level 3

| Level | 狀態 | 特徵 |
|-------|------|------|
| Level 1 | 一個 .py | 直接 `python xxx.py` 執行 |
| Level 2 | + `if __name__` | 被 import 時底部不會跑 |
| Level 3 | + `__init__.py` = Package | 有結構、方便 import、可安裝 |

## FAQ：WSL 問題集

### Q1：wsl --install 出現錯誤 0x80370102

**原因**：BIOS 虛擬化沒開。

**解法**：
1. 重新啟動電腦，開機時按 DEL / F2 / F10（依主機板品牌不同）進入 BIOS
2. 找到以下其中一項並啟用：
   - Intel：**Intel VT-x** 或 **Intel Virtualization Technology**
   - AMD：**AMD-V** 或 **SVM Mode**
3. 存檔退出 → 重開機
4. 重新執行 `wsl --install`

**怎麼確認虛擬化有開**：工作管理員 → 效能 → CPU → 右下角「虛擬化：已啟用」

### Q2：wsl -l -v 顯示 VERSION 是 1 不是 2

**解法**：

```powershell
# 轉換為 WSL2
wsl --set-version Ubuntu-22.04 2

# 等待轉換完成（可能需要幾分鐘）

# 設定以後都預設 WSL2
wsl --set-default-version 2

# 再次確認
wsl -l -v
# VERSION 要是 2
```

**如果轉換失敗**：

```powershell
# 用 PowerShell（系統管理員）啟用虛擬化功能
dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart
dism.exe /online /enable-feature /featurename:Microsoft-Windows-Subsystem-Linux /all /norestart

# 重新啟動電腦
# 再跑 wsl --set-version Ubuntu-22.04 2
```

### Q3：Windows 11 Home 版找不到 Hyper-V

**答**：不需要 Hyper-V。WSL2 只需要「虛擬機器平台」（Virtual Machine Platform），`wsl --install` 會自動啟用。

- Hyper-V = 完整虛擬機管理（Home 版沒有）
- Virtual Machine Platform = 輕量虛擬化（Home 版有）
- WSL2 只需要後者

### Q4：Ubuntu 打不開 / 卡住

**解法**：

```powershell
# Step 1：重啟 WSL
wsl --shutdown
# 等 5 秒，再開 Ubuntu

# Step 2：如果還是不行
wsl --update
wsl --shutdown
# 再開

# Step 3：最後手段（⚠️ 會刪除所有資料）
wsl --unregister Ubuntu-22.04
wsl --install -d Ubuntu-22.04
```

### Q5：登入是 root 不是個人帳號

**解法**：

```powershell
# 在 Windows PowerShell 執行
ubuntu2204 config --default-user 你的帳號
```

如果還沒建過帳號：

```bash
# 在 WSL 裡（root 身份）
adduser 你的帳號
usermod -aG sudo 你的帳號
```

然後再回 PowerShell 設定 `ubuntu2204 config --default-user 你的帳號`

### Q6：Microsoft Store 找不到 Ubuntu 22.04

**解法**：用指令安裝

```powershell
wsl --install -d Ubuntu-22.04
```

這會直接從微軟伺服器下載，不經過 Store，效果完全一樣。

### Q7：已經裝了 Ubuntu 18.04 怎麼辦？

VS Code Remote WSL 要求 glibc ≥ 2.28，Ubuntu 18.04 的 glibc 是 2.27 → VS Code Server 無法啟動。

**解法**：再裝 22.04，不用移除 18.04

```powershell
wsl --install -d Ubuntu-22.04
wsl --set-default Ubuntu-22.04
```

想移除 18.04（⚠️ 會刪除所有資料）：

```powershell
wsl --unregister Ubuntu-18.04
```

### Q8：WSL 裡的檔案存在 Windows 的哪裡？

WSL 的檔案系統位於 `\\wsl$\Ubuntu-22.04\` 底下，可以在 Windows 檔案總管輸入這個路徑存取。但建議不要從 Windows 直接修改 WSL 裡的檔案，效能和權限都會有問題。

### Q9：WSL 和 Windows 怎麼互相存取檔案？

```bash
# 在 Windows 存取 WSL 檔案
# 檔案總管輸入：\\wsl$\Ubuntu-22.04\home\你的帳號\
```

### Q10：sudo apt update 很慢或失敗

**解法**：

```bash

sudo apt update
```

**答**：Linux 的密碼輸入不會顯示任何字元（連星號都沒有），這是安全設計。直接打完密碼按 Enter 就好。

**解法**：確認 Windows 版本夠新

### Q13：WSL 開不了，顯示 "WslRegisterDistribution failed"

### Q14：WSL 佔太多記憶體

**解法**：建立 `.wslconfig` 限制資源

```powershell

```

內容：

```ini

swap=1GB
```

設定完執行 `wsl --shutdown`，再重新開啟。

### Q15：PowerShell 顯示 "wsl 不是內部或外部命令"

**答**：Windows 版本太舊，不支援 WSL。需要 Windows 10 version 1903+ 或 Windows 11。執行 Windows Update 更新到最新版。

### Q16：wsl --install 卡在 "Installing, this may take a few minutes..."

**答**：網路問題。確認網路連線穩定，等待下載完成。如果等超過 10 分鐘沒動靜，按 Ctrl+C 中斷，再跑一次。

### Q17：已安裝的 Ubuntu 版本怎麼看？

```bash

```

```powershell

```


### Q1：code . 沒反應

**Windows**：重裝 VS Code，安裝時確認勾選 "Add to PATH"。或者手動把 VS Code 加到環境變數。

**Mac**：在 VS Code 裡 Cmd+Shift+P → 搜尋 "shell command" → 選 "Install 'code' command in PATH"

### Q2：VS Code 左下角不是 WSL: Ubuntu

**解法**：
1. 確認已安裝 WSL Extension
2. 在 Extensions 搜尋 "WSL"（Microsoft 官方），按 Install
3. 從 WSL Terminal 執行 `code .`，VS Code 會自動連線
4. 如果還是不行：Ctrl+Shift+P → "Remote-WSL: New Window"

### Q3：VS Code Server for WSL closed unexpectedly

**逐步排除**：

```bash
# Step 1：重啟 WSL
# 在 PowerShell：
wsl --shutdown
# 等 5 秒，再開 Ubuntu，跑 code .

# Step 2：刪 Server 重裝
# 在 WSL Ubuntu terminal（不是 PowerShell）：
rm -rf ~/.vscode-server
# 再跑 code .（會自動重新下載）

# Step 3：更新
# PowerShell：
wsl --update
wsl --shutdown
# 重開 VS Code

# Step 4：更新 VS Code 和 Extension
# VS Code 齒輪 → Check for Updates
# WSL Extension 確認沒有 Update 按鈕

# Step 5：查 log 找老師
cat ~/.vscode-server/data/logs/*/remoteagent.log | tail -20
# 截圖交老師或 TA
```

### Q4：之前裝的套件和 Python 版本不見了

**答**：因為現在在 WSL Ubuntu 環境裡，跟 Windows 是分開的。Windows 裝的 Python 和套件在 WSL 裡看不到。需要在 Ubuntu 裡重新安裝：

```bash
sudo apt install python3-pip -y
pip3 install requests beautifulsoup4 pandas
```

### Q5：VS Code Terminal 是 PowerShell 不是 WSL

**解法**：
1. 點 Terminal 右上角的下拉選單
2. 選 "Ubuntu (WSL)" 或 "bash"
3. 或者設定預設 Shell：
   - Ctrl+Shift+P → "Terminal: Select Default Profile" → 選 "Ubuntu (WSL)"

### Q6：VS Code 自動完成不工作

**解法**：
1. 確認 Python Extension 已安裝
2. Ctrl+Shift+P → "Python: Select Interpreter"
3. 選擇正確的 Python 版本（WSL 裡的 python3）

### Q7：VS Code 存檔時出現權限錯誤

**解法**：
1. 確認檔案不是 root 擁有的：`ls -la 檔案名`
2. 如果是：`sudo chown $USER:$USER 檔案名`
3. 確認目錄有寫入權限：`chmod 755 目錄名`

### Q8：VS Code 卡頓 / 很慢

**解法**：
1. 停用不需要的 Extension
2. 關閉其他佔記憶體的程式
3. 如果在 WSL，確認 `.wslconfig` 有給足記憶體

### Q9：VS Code 中文亂碼

**解法**：
1. File → Preferences → Settings
2. 搜尋 "encoding"
3. 設定 `Files: Encoding` 為 `utf-8`

### Q10：VS Code 無法開啟大檔案

**答**：VS Code 預設限制開啟大檔案（>50MB）。可以在 Settings 裡搜尋 `files.maxMemoryForLargeFilesMB` 增加限制。

### Q11：VS Code Remote 連線斷開後重連失敗

```bash
# 在 WSL 裡清除快取
rm -rf ~/.vscode-server/bin/*
# 再跑 code .
```

### Q12：VS Code Extension 裝了但沒效果

**答**：WSL 環境的 Extension 要在 WSL 側安裝。打開 Extensions 面板，找到 Extension 後看是否顯示 "Install in WSL"。部分 Extension 需要在 WSL 側重新安裝。

## FAQ：uv 與 Python 環境問題集

### Q1：uv: command not found

**解法**：

```bash
# 重新安裝
curl -LsSf https://astral.sh/uv/install.sh | sh

# 重新載入 shell
source ~/.bashrc
# 或
exec $SHELL

# 確認
uv --version
```

### Q2：uv sync 失敗

**常見原因**：

```bash
# 1. 沒有 pyproject.toml
ls pyproject.toml
# 如果不存在 → 確認你在正確的目錄

# 2. 網路問題
# 確認網路正常：ping pypi.org

# 3. Python 版本不符
cat .python-version
# 確認有對應的 Python 版本
uv python list
```

### Q3：import 找不到套件（ModuleNotFoundError）

**解法**：用 `uv run python xxx.py` 而不是 `python3 xxx.py`。

`uv run` 會自動使用虛擬環境裡的 Python 和套件。直接用 `python3` 是用系統 Python，不包含 uv 管理的套件。

### Q4：uv add 報錯 "No project found"

**解法**：確認當前目錄有 `pyproject.toml`。如果沒有，先初始化：

```bash
uv init
# 或 uv init my-project
```

### Q5：uv run 報 "No .venv found"

```bash
# 建立虛擬環境
uv venv
# 然後再 uv sync
uv sync
```

### Q6：pip install 和 uv add 有什麼不同？

| | `pip install` | `uv add` |
|--|--------------|---------|
| 裝到哪 | 全域或手動 activate 的 venv | 自動裝到 .venv |
| 記錄 | 不記錄（要手動 freeze） | 自動加到 pyproject.toml |
| 鎖版 | 不鎖（requirements.txt 手動維護） | 自動產生 uv.lock |
| 速度 | 普通 | 極快（10-100x） |

### Q7：uv 和 pip 可以混用嗎？

**不建議**。混用會導致 `pyproject.toml` 和實際安裝不一致。如果用 uv 管理專案，所有套件操作都用 uv。

### Q8：怎麼看 uv 專案裝了哪些套件？

```bash
# 看 pyproject.toml 裡的 dependencies
cat pyproject.toml

# 看依賴樹
uv tree

# 看 lock 檔（精確版本）
cat uv.lock
```

### Q9：uv run python 和直接 python3 結果不同

**答**：`uv run python` 使用虛擬環境裡的 Python（有 uv 管理的套件），`python3` 使用系統 Python（沒有那些套件）。在 uv 專案裡永遠用 `uv run python`。

### Q10：怎麼移除 uv 安裝的套件？

```bash
uv remove requests
# 自動從 pyproject.toml 移除 + 更新 uv.lock
```

### Q11：pyproject.toml 和 requirements.txt 的差別？

| | `requirements.txt` | `pyproject.toml` |
|--|-------------------|-----------------|
| 格式 | 純文字清單 | TOML 結構化 |
| 資訊量 | 只有套件名和版本 | 包含專案名、版本、Python 版本要求 |
| 鎖版 | 沒有（或搭配 requirements.lock） | 搭配 uv.lock |
| 標準 | 非官方標準 | PEP 621 官方標準 |
| 支援工具 | pip | uv, poetry, pdm, flit |

### Q12：版本衝突（version conflict）怎麼解？

```bash
# 先嘗試升級 lock 檔
uv lock --upgrade

# 如果還是不行，看衝突細節
uv tree
# 找出哪兩個套件要求不同版本的同一個依賴
```
